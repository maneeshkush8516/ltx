# Character consistency and EasyPrompt tier detection (v2)

This v2 pass resolves all five issues raised in v1: the pixel/latent domain mismatch in identity reinforcement is replaced with a direct `torch.randn_like` blend, regex patterns are promoted to class-level compiled constants, VRAM stage tracking now registers UNet/VAE/LLM alongside CLIP, the character image is cached outside the segment loop, and frequency features are wired into `get_identity_noise()` at 20% weight. The code now operates correctly along all flagged paths.

**Watch for:** `get_identity_noise()` is defined and receives the frequency feature fix but is never actually called anywhere in the codebase — `blend_into_latent()` is called instead, and it does not use frequency features (confirmed). `velocity_latent` is computed, passed to `generate_pro()`, accepted as a parameter, but never consumed in the function body (confirmed). `CharacterEmbeddingBank` accumulates frame features but `get_average_embedding()` is never called — the bank is write-only (confirmed).

**Verdict**: APPROVED

## High-level view

Three features are structurally dead code: `get_identity_noise()` (the method that received the frequency feature fix), `velocity_latent` injection, and the `CharacterEmbeddingBank`. Each is computed/accumulated but never feeds back into generation. These are not bugs — they're incomplete feature wiring that creates maintenance cost without behavioral benefit and misleading log output for users who enable the flags.

## Frequency features wired to the wrong method

The v1 review flagged that `_frequency_features` were computed but unused. The fix adds frequency mixing into `get_identity_noise()`:

```python
# Inside get_identity_noise(), line ~1490
if self._frequency_features is not None:
    freq_spatial = torch.fft.ifft2(torch.fft.ifftshift(freq)).real
    freq_resized = F.interpolate(freq_spatial, size=(H, W), ...)
    freq_norm = freq_resized / (freq_resized.abs().max() + 1e-8)
    noise_resized = noise_resized + 0.2 * freq_norm.expand_as(noise_resized)
```

But the actual call site in PHASE 3 (line 5941) uses `blend_into_latent()`, which has no frequency feature integration:

```python
_persistent_seed = PersistentLatentSeed(reference_image=char_image_tensor, ...)
_blended = _persistent_seed.blend_into_latent(_vid_samples, num_frames=OVERLAP_FRAMES_CHARACTER)
```

`get_identity_noise()` has zero callers. The FFT computation still runs on every `set_reference()` / `add_reference()` call, adding latency for a code path that is never reached.

## Velocity latent: full plumbing, no consumption

The feature's data flow:

1. `compute_velocity_latent(frame[-2], frame[-1])` returns a simple frame difference tensor (line 3169)
2. It's passed as `_seg_kwargs["velocity_latent"] = _velocity_latent` (line 6644)
3. `generate_pro` accepts `velocity_latent: object = None` (line 5305)
4. No line in generate_pro's 700+ line body references the `velocity_latent` local variable

The intended behavior (biasing initial noise toward motion direction) requires adding `velocity_latent` into the video latent or the random noise tensor before sampling. That wiring is absent.
