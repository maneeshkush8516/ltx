# Character consistency strengthening and EasyPrompt tier detection port

This change enhances a single-file Google Colab video generation script (~8k lines) with two feature clusters: (1) multi-frame character identity preservation via upgraded PersistentLatentSeed, CharacterFeatureExtractor, and segment-boundary reinforcement; (2) full content-tier detection logic ported from a sibling script, plus VRAM stage tracking, overlap frame accounting, story-to-prompt character extraction, and audio sync stubs. The approach threads new logic into an existing monolith without restructuring it, relying on defensive try/except wrappers and print-based telemetry.

**Watch for:** Identity reinforcement operates on pixel-space anchor frames but passes them through `blend_into_latent()` which assumes NCHW layout when the input is NHWC (confirmed). The `_verify_vram_clear` helper raises `RuntimeError` on VRAM threshold violation, which is caught locally but could surface in unexpected contexts if the pattern is extended. Character profile extraction mutates a module-level class attribute (`InlineVisionDescribe.DESCRIBE_PROMPT`) without thread safety (confirmed). The full SYSTEM_PROMPT now contains explicit adult content instructions permanently baked into the prompt template regardless of input content (confirmed).

## High-level view

PersistentLatentSeed was rewritten from a simple noise blender into a multi-reference class with frequency-domain feature extraction and a temporal-decay `get_identity_noise()` method. The FFT features are computed but never consumed downstream — they sit in `_frequency_features` unused. The integration into `generate_pro()` PHASE 3 and `generate_extended_video()` segment boundaries is wrapped in try/except so failures are non-fatal, which is appropriate for a Colab notebook context.

The InlinePromptArchitect.generate() method tripled in size by importing the full regex-based content-tier detection stack from LTX2EasyPromptLD.py. All regex patterns are compiled inside the method body on every call rather than as class/module constants, creating unnecessary per-call overhead for patterns that never change.

VRAMManager gains stage tracking (register/release) and `suggest_offload_order()`, but only CLIP is registered; the UNet and VAE loads don't call `register_stage()`, so the tracking map is always incomplete. The `_verify_vram_clear` helper provides a hard gate between model transitions on T4, which is a genuine improvement for the target hardware.

The identity reinforcement in `generate_extended_video()` blends noise into the last-frame anchor *in pixel space* then saves it as PNG. The `blend_into_latent()` method's dimension-detection heuristic (`shape[-1] <= 4` means NHWC) will misfire on an NHWC (1, H, W, 3) tensor because `ndim == 4` and `shape[-1] == 3 <= 4`, causing a permutation to NCHW before the blend. The resulting tensor is then passed to `tensor_to_pil()` which expects NHWC, producing a corrupted anchor image.

The overlap frame accounting in the stitching phase prints an expected total but doesn't actually change the stitching logic — it's a diagnostic comment with no behavioral impact on how frames are blended.

## Pixel-space blend mismatch in identity reinforcement

In `generate_extended_video()` around line 6690, the code calls:

```python
if last_frame_tensor.ndim == 4:
    _reinforced = _reinforce_seed.blend_into_latent(last_frame_tensor)
    _reinforced_pil = tensor_to_pil(_reinforced)
```

`last_frame_tensor` is produced by `get_last_frame_tensor()` which returns shape `(1, H, W, 3)` — a pixel-space NHWC tensor with values in [0, 1]. Inside `blend_into_latent`, the 4D branch executes:

```python
if noise.ndim == 4 and noise.shape[-1] <= 4:
    noise = noise.permute(0, 3, 1, 2)  # NHWC -> NCHW
```

This fires because `3 <= 4`. The noise becomes NCHW (1, 3, H, W). The blend then produces an NCHW tensor. But `tensor_to_pil()` expects NHWC and does `t[0].cpu().numpy() * 255` — it would interpret the 3-element channel dim as height (3 rows), producing a corrupt 3xW image or an error.

The same heuristic is safe in `generate_pro()` PHASE 3 because there the input is an actual latent `(B, C, T, H, W)` with `ndim == 5`, taking the 5D branch instead.

This is a confirmed bug that will produce a garbled anchor PNG at every segment boundary when `USE_IDENTITY_REINFORCEMENT` is True.

## Dead frequency-domain computation

`_compute_frequency_features()` runs FFT on the reference image, applies a low-pass mask, and stores the result in `self._frequency_features`. Neither `blend_into_latent()`, `get_identity_noise()`, nor `get_noise()` reads this field. The docstring says "for more stable identity preservation" but the feature is aspirational, not wired. The FFT computation adds latency on every `set_reference()` / `add_reference()` call for no benefit.

## Regex compilation inside the hot path

The `generate()` method now compiles 6+ regex patterns (`_explicit_re`, `_sensual_re`, `_undress_re`, `_already_naked_re`, `_clothing_re`, `_mid_action_re`, `_person_re`, `_multi_re`, `_static_re`, etc.) on every call. In a storyboard workflow, `generate()` is called per scene — potentially dozens of times in a single run. These patterns are static and should be class-level `re.compile()` constants.

## Incomplete stage tracking defeats offload ordering

`suggest_offload_order()` returns a priority list from `self._loaded_stages`. Only one call site registers a stage:

```python
_VRAM_MGR.register_stage("CLIP", 10.0 if _VRAM_MGR.is_t4 else 8.0)
```

UNet loading (PHASE 3), VAE loading (audio and video), and the LLM model load in `InlinePromptArchitect._load_model()` never register. The function will always return `["CLIP"]` at most, making the offload priority system non-functional in practice.

## Module-level attribute mutation for profile extraction

```python
_original_prompt = InlineVisionDescribe.DESCRIBE_PROMPT
InlineVisionDescribe.DESCRIBE_PROMPT = self.CHARACTER_EXTRACT_PROMPT
try:
    raw_output = vision_desc.describe(...)
finally:
    InlineVisionDescribe.DESCRIBE_PROMPT = _original_prompt
```

This mutates a class attribute on a shared class. In a Colab notebook this is unlikely to race, but if `describe()` raises before entering the try block (impossible here but fragile as a pattern), or if the notebook uses threaded cell execution, the prompt stays mutated. A safer pattern would pass the prompt as a parameter to `describe()`.

## Overlap frame accounting is diagnostic-only

The added code:

```python
_expected_total = sum(segment_lengths[:len(all_segment_paths)]) - max(0, len(all_segment_paths) - 1) * overlap_frames
print(f"   Expected total frames after overlap subtraction: ~{_expected_total}")
```

This prints the corrected total but doesn't feed it into any logic. The actual stitching loop below iterates `all_segment_paths` and blends based on the existing `overlap_frames` parameter unchanged. The accounting is correct as documentation, but if the user expects the output duration to match `_expected_total`, it won't unless the blending code downstream already handles this (which it does — the blend_overlap_frames function trims overlap). The comment could mislead into thinking something was fixed when the calculation was already implicit.

## Character extraction from scripts — shallow heuristic

`extract_character_profiles_from_script()` identifies characters by finding capitalized proper nouns that appear 2+ times, filtered against a hardcoded stop-word list. This will false-positive on place names (Paris, Tokyo), brands, and any capitalized noun. The profile generated for detected names is just context sentences from the script, not a structured CharacterFeatureExtractor profile. The function name suggests LLM-based extraction but it's regex-only, which is fine for a Colab context but the gap between docstring promise and implementation is worth noting.
