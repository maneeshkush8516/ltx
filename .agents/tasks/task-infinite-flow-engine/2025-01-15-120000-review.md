# Infinite Flow Engine v2.0 — scaffolding and inline model integration

The diff restructures LTX_PRO.py from a v1.0 pipeline into a "v2.0 Infinite Flow Engine" by adding ~1160 lines: VRAMManager for GPU detection, InlinePromptArchitect/InlineVisionDescribe as fallback prompt expansion paths, PersistentLatentSeed/CharacterPromptAnchor for character consistency, AudioSyncEngine as a placeholder, FlowState/InfiniteFlowConfig dataclasses, and a `generate_infinite_flow()` entry point that wraps `generate_extended_video()`. The approach is additive — new classes are defined alongside the existing pipeline but mostly not wired into the execution path.

**Watch for:** `generate_infinite_flow()` is never called by Cell 9, so the headline feature is dead code (confirmed). `PersistentLatentSeed.blend_into_latent` is a no-op that returns its input unchanged (confirmed). The inline prompt architect strips only ~30% of the cleanup patterns the original `LTX2EasyPromptLD.py` handles, meaning LLM output will leak compliance spam in fallback mode (confirmed). Several new classes (`VRAMManager`, `AudioSyncEngine`, both character classes) are defined but never referenced by the generation pipeline (confirmed).

## High-level view

`generate_infinite_flow()` is a thin facade over `generate_extended_video()`, adding FlowState tracking and InfiniteFlowConfig support. Cell 9's run logic still dispatches to `generate_extended_video()` directly when `USE_SEGMENT_EXTENSION=True`, making the new entry point unreachable. FlowState fields (anchor frame, embedding bank, quality scores, segment paths) are never populated during generation — only `segment_index` and `total_frames` get rough estimates post-run.

InlinePromptArchitect condenses 600+ lines of `LTX2EasyPromptLD.py` into a minimal fallback. The system prompt is drastically simplified — the full source has content-tier detection (explicit/sensual/neutral), undressing sequence instructions, and detailed pacing enforcement that the inline version omits. The `_clean_output` method retains only 10 of ~30 regex cleanup passes, leaving the fallback path vulnerable to compliance checklist spam, filler-character repetition, and leaked instruction labels.

VRAMManager detects GPU memory and categorizes into T4/L4/A100 tiers, but `get_optimal_settings()` output is never consumed by `generate_pro()` or any other function. The `vram_guard` decorator is defined but never applied.

PersistentLatentSeed's `blend_into_latent` — the method that would inject character noise into the generation latent — returns its input unchanged. CharacterPromptAnchor is functional in isolation but never instantiated or called from the pipeline.

AudioSyncEngine is a pure placeholder with no callers.

## `generate_infinite_flow` delegates without adding behavior

The function resolves parameters from `globals()`, constructs an `InfiniteFlowConfig`, then passes everything through to `generate_extended_video`. The only new behavior is:

```python
if result:
    flow_state.segment_index = config.max_segments
    flow_state.total_frames = config.max_segments * config.segment_length
```

This assumes all segments completed successfully — no partial-failure handling. The docstring promises "segment-level prompt variation and continuation prompt generation" but neither is implemented.

Cell 9 at line 6056 still routes to `generate_extended_video()`. The usage instructions in Cell 11 tell users that `USE_SEGMENT_EXTENSION=True` "uses generate_infinite_flow()" — this is incorrect.

## InlinePromptArchitect — incomplete cleanup

The `_clean_output` method handles 10 patterns. Missing from the original's 30+:

- Instruction label stripping (`Action Beat 1:`, `Undressing Segment:`, etc.)
- Orphaned closing bracket spam
- Fake conversation/self-eval patterns (`Please let me know`, `Let me revise`, etc.)
- Model loop/panic patterns (`Ended. 5 actions`, `Done. 6 seconds`, etc.)
- Filler character spam (repeated single characters)
- Leaked pacing instruction echoes
- Token/word count lines
- Screenplay-style bracketed camera directions (`(DOWN 10°)`, `(HOLD)`, etc.)
- `(Lora:...)` tag echoes
- Timestamp annotations

The `_build_negative_prompt` also drops `_NEG_EXPLICIT` (censored/mosaic), `_NEG_WIDE` (close-up crop), and `_NEG_MULTI` (merged bodies) categories present in the original.

## PersistentLatentSeed — hash collision risk and dimension errors

`set_reference` hashes image content via `hash(img_data.sum().item())`. Since `.sum()` collapses all spatial information into a single scalar, visually different images with similar pixel sums produce identical hashes and identical noise tensors — defeating per-character noise generation.

When the reference is not a `torch.Tensor`, `img_data` falls back to `torch.zeros(1)`. If `get_noise` is later called with a `target_shape`, `F.interpolate` will attempt to resize a 1-element tensor to a 4D shape, raising a dimension error.

The `get_noise` resize logic assumes NHWC layout when `noise.shape[-1] <= 4` and permutes accordingly. This heuristic breaks for video diffusion latents where channel dimensions are 16 or 32.
