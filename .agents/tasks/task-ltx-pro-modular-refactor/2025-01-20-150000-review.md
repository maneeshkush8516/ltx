# Modular extraction of LTX_PRO into ltx_pro/ package with Story-to-Video feature

The 8670-line monolithic Colab notebook has been split into an 18-module Python package (`ltx_pro/`) and a 719-line thin orchestrator (`LTX_PRO.py`). The orchestrator retains all 134 `@param` annotations for Colab widget rendering while delegating implementation to domain-specific modules. A new `StoryToVideo` class adds YouTube series production capabilities (character profiling, chapter markers, recap scenes, episode continuity).

**Watch for:** The extracted modules call `globals()` expecting to find Colab-level config variables (`SHOW_PREVIEWS`, `ENABLE_*`, `VAE_VIDEO_MODEL`, etc.), but `globals()` now returns each module's own namespace where those variables don't exist â€” every such lookup silently falls back to hardcoded defaults (confirmed). The `from ltx_pro.config import *` in the orchestrator populates LTX_PRO.py's namespace but is invisible to pipeline.py, extended.py, and storyboard.py at runtime. This breaks feature toggles, Google Drive sync, timeline export, and quality gate activation.

## High-level view

The refactoring divides the codebase into four tiers: foundation modules (config, vram, utils) that have no cross-dependencies within the package; domain modules (prompt_architect, vision_describe, character, lora, overlap, quality, motion, color, audio, export) each encapsulating one functional concern; pipeline orchestration modules (pipeline, extended, storyboard) that compose the domain modules into generation workflows; and a new Story-to-Video module that layers YouTube series logic on top of the storyboard pipeline.

The central architectural coupling problem is that `generate_pro()`, `generate_extended_video()`, `generate_infinite_flow()`, and `run_storyboard()` all call `_g = globals()` inside their function bodies and use `_g.get("VARIABLE_NAME", default)` to read approximately 40 configuration values. In the original monolith, those variables lived in the same module scope. Now they live in `LTX_PRO.py` (or nowhere, if config.py's star-import semantics differ from runtime expectations). The result is that user-configured values in the Colab UI are never propagated to the actual generation logic.

The new `StoryToVideo` class is a well-isolated addition: it imports only from `ltx_pro.storyboard`, `ltx_pro.config`, `ltx_pro.export`, and `ltx_pro.character`, avoiding the globals coupling entirely by passing data through method arguments and constructor config.

## globals() namespace mismatch across module boundaries

In the original monolith, `generate_pro()` was defined in the same file where all config variables (`SHOW_PREVIEWS`, `DOWNLOAD_AFTER_GENERATE`, `VAE_VIDEO_MODEL`, `ENABLE_*` flags, etc.) were declared. Calling `globals()` returned that single shared namespace. After extraction into `ltx_pro/pipeline.py`, the function's `globals()` returns `pipeline.py`'s module dict, which contains only `gc`, `os`, `time`, `json`, and the `TYPE_CHECKING` guard.

The practical impact:

```python
# pipeline.py line 288
_g = globals()

# These always return their default fallback:
_g.get("SHOW_PREVIEWS", False)          # always False
_g.get("DOWNLOAD_AFTER_GENERATE", False) # always False
_g.get("VAE_VIDEO_MODEL", "")           # always ""
_g.get("ENABLE_UPSCALING", True)        # always True (cannot be disabled)
_g.get("ENABLE_CHARACTER_SYSTEM", True)  # always True
_g.get("GDRIVE_PATH", "")              # always ""
```

The same pattern affects `extended.py` (line 122) and `storyboard.py` (lines 128, 315, 424). Every function that relied on Colab-level globals now operates on phantom defaults.

The fix with lowest surface area: add `from ltx_pro.config import *` at the top of `pipeline.py`, `extended.py`, and `storyboard.py`. This propagates config.py's star-exported names into each module's globals. However, this only solves half the problem â€” variables defined solely in LTX_PRO.py (like `SHOW_PREVIEWS`, `CHARACTER_IMAGE_PATH`, `GDRIVE_PATH`) are not in config.py's `__all__` and won't be propagated. A config dataclass passed as an argument is the cleaner solution.

## apply_preset() vs Colab @param execution order

In `config.py`, `apply_preset()` mutates module-level globals:

```python
def apply_preset():
    global WIDTH, HEIGHT, FRAMES, LLM_MODEL, ...
    preset = _PRESET_DEFINITIONS.get(GENERATION_PRESET)
    if preset:
        WIDTH = preset["width"]
        ...
```

In the orchestrator, Cell 3.5 calls `apply_preset()` after importing config's star exports. But Cell 6 then re-declares `WIDTH = 768 # @param {type:"integer"}`. Colab `@param` cells execute their assignments unconditionally â€” the user sees a widget pre-filled with 768, and running the cell sets `WIDTH = 768` regardless of what `apply_preset()` set. The preset system is therefore inert unless the user manually deletes the Cell 6 assignments, which defeats the purpose.

## StoryToVideo as self-contained addition

The `StoryToVideo` class (823 lines) introduces a YouTube-oriented workflow: load a script, extract character profiles via regex heuristics, decompose into scenes using the existing `decompose_script_to_scenes()`, add cliffhanger detection, generate recap scenes, produce chapter markers, and export a production package (timeline, EDL, character sheets, Sora JSON).

The class is well-isolated â€” it imports only from other `ltx_pro` submodules and does not use `globals()`. Character extraction relies on regex patterns matching near character names (age, hair, clothing, body type, distinguishing features). The regex approach is lightweight but fragile for narratives that describe characters indirectly or use non-English naming conventions. The cliffhanger detection uses keyword matching against a 30-word list, which may produce false positives on action-heavy scenes that aren't intended as episode endings.

The `generate_all_videos()` method delegates to `run_storyboard()`, which inherits the globals mismatch problem â€” so while the StoryToVideo class itself is clean, its execution path passes through broken infrastructure.

## Colab-specific coupling remains in the package

Several modules import `google.colab.files` or `IPython.display` inside function bodies (pipeline.py, utils.py, export.py). These are guarded by try/except, which is the standard Colab pattern. However, the package advertises itself as importable (`__version__ = "2.0.0"`) while being structurally dependent on Colab runtime conventions (hardcoded paths like `/content/ComfyUI/output/`, `%cd` magics assumed in the orchestrator, Google Drive mount paths). This is acceptable for the stated use case (Colab notebook) but means the package cannot be used outside Colab without modification.

## config.py star-export surface

`config.py`'s `__all__` exports 97 names including both constants and functions. The `from ltx_pro.config import *` in the orchestrator dumps all of these into the Colab cell's namespace. Since the orchestrator immediately re-declares most of these same names with `@param` annotations (e.g., `WIDTH = 768 # @param`), the config.py values are overwritten. The only config.py exports that survive are the helper functions (`apply_preset`, `validate_config`, `print_current_config`, etc.) and constants not re-declared in the orchestrator.
