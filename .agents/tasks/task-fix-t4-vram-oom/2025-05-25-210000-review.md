# Aggressive VRAM cleanup to fix T4 OOM

The change solves CUDA out-of-memory failures on T4 GPUs (16 GB) by ensuring LLM and Vision models are fully evicted from VRAM before CLIP and UNet are loaded. The root cause is that HuggingFace `accelerate`'s `device_map="auto"` attaches dispatch hooks that hold CUDA tensor references, making `.to("cpu")` + `del` insufficient. The fix adds hook removal, explicit per-parameter CPU migration, triple `gc.collect()`, `ipc_collect()`, verification checkpoints, and a `force_unload_all_models()` nuclear fallback that walks the entire Python object graph. A new `VRAMManager.enforce_sequential_loading()` method gates each pipeline phase on a minimum available-VRAM threshold.

**Watch for:** `force_unload_all_models()` iterating `gc.get_objects()` is O(all-Python-objects) and may stall on large heaps (confirmed); the `get_available_vram()` calculation uses only `memory_allocated`, not `memory_reserved`, so the threshold check can pass while the CUDA allocator still holds reserved blocks that prevent new allocations (confirmed); identical 30-line unload sequences are copy-pasted four times with no shared implementation (confirmed).

## High-level view

The unload logic in all four model-hosting sites (standalone `LTX2EasyPromptLD`, standalone `LTX2VisionEasyPromptLD`, inline `InlinePromptArchitect`, inline `InlineVisionDescribe`) is extended with the same three-step protocol: remove `accelerate` dispatch hooks, iterate every parameter/buffer and move it to CPU individually, then run an amplified cleanup sequence. This is repeated verbatim in each location — a maintenance hazard if the protocol ever needs updating.

A new `force_unload_all_models()` function acts as a safety net: it walks `gc.get_objects()`, finds any `PreTrainedModel` with CUDA parameters, strips its hooks, and moves its tensors to CPU. It is called unconditionally in the `generate_pro` pipeline between Phase 0 (LLM/Vision) and Phase 1A (CLIP), even when normal unloading already succeeded.

`VRAMManager.enforce_sequential_loading()` gates model loading on hard VRAM thresholds (12 GB free for CLIP on T4, 8 GB elsewhere; 6 GB for UNet on T4). If cleanup cannot meet the threshold, it raises a `RuntimeError` with diagnostic context rather than letting the process OOM deep inside ComfyUI's loader.

<details>
<summary>Issues (5)</summary>

1. **gc.get_objects() scaling** — `force_unload_all_models()` iterates all tracked Python objects on every call. In a ComfyUI server process with many loaded nodes this can take hundreds of milliseconds. Extract it behind a flag or call it only in the retry path, not as the first-pass strategy.
2. **memory_allocated vs memory_reserved** — `get_available_vram()` subtracts only `torch.cuda.memory_allocated()` from total VRAM. The CUDA allocator may still reserve blocks that prevent new allocations. Use `torch.cuda.mem_get_info()[0]` for a physical-free measurement from the driver.
3. **Duplicated unload protocol** — The hook-removal + per-param-move + triple-gc sequence is copy-pasted in four locations across three files. Extract into a shared helper (e.g. `deep_unload(model)`) so future fixes apply once.
4. **Hard-coded VRAM thresholds** — 12 GB "free" for CLIP on T4 is close to the total 15.9 GB device memory, leaving <4 GB for PyTorch's own CUDA context + reserved pools. If any framework state is resident, this will always raise RuntimeError. Consider deriving the threshold from actual model size or using a percentage of total.
5. **time.sleep(0.1) on critical path** — Each unload path includes a 100 ms sleep, and the pipeline executes up to three cleanup rounds. Worst case adds ~0.5 s of idle time on every generation, even when VRAM is already clean.

## force_unload_all_models and gc.get_objects scaling

`force_unload_all_models()` iterates `gc.get_objects()` — the entire set of garbage-collector-tracked Python objects — and type-checks each one against `PreTrainedModel`. In a typical ComfyUI process with custom nodes, large prompt queues, and cached model references, this set can contain millions of objects. The function is called unconditionally in the happy path (before CLIP load) and potentially called three more times in the retry branches.

The function silently swallows all exceptions per model (`except Exception: pass`), which means a partially-moved model (some params on CPU, some still on CUDA) will be counted as "unloaded" and won't be retried. If one param move raises (e.g., due to a storage-sharing tensor), the remaining CUDA tensors in that model stay pinned.

## VRAM threshold check uses memory_allocated, not physical free

```python
def get_available_vram(self):
    allocated = torch.cuda.memory_allocated() / 1024**3
    return self.total_vram_gb - allocated
```

After `empty_cache()`, reserved should equal allocated, so this is often correct post-cleanup. But CUDA context memory (300–800 MB on T4), cuBLAS workspaces, and any other non-PyTorch CUDA allocations won't appear in `memory_allocated()`. `torch.cuda.mem_get_info()` returns `(free, total)` from the driver and would give ground-truth available memory.

The threshold of 12 GB for CLIP on a 15.9 GB T4 means the check passes only if ≤3.9 GB is used by everything else. This is tight — the CUDA context alone consumes 500+ MB. If it proves too aggressive, the `RuntimeError` will fire on every T4 run, which is a better failure mode than OOM but the user may not be able to resolve it without reducing the threshold or switching to `mem_get_info`.

## Duplicated unload sequences

The same ~30-line block (hook removal → `_hf_hook` deletion → param iteration → `.to("cpu")` → triple gc → synchronize → empty_cache → ipc_collect → sleep → gc → empty_cache → print) appears in:

- `LTX2EasyPromptLD.py` → `unload_model()`
- `LTX2VisionEasyPromptLD.py` → inline unload block
- `LTX_PRO.py` → `InlinePromptArchitect._unload_model()`
- `LTX_PRO.py` → `InlineVisionDescribe._unload()`

The only differences are log prefixes and whether exceptions are printed or silently swallowed. If the cleanup protocol needs to change (e.g., to handle a new accelerate version's hook structure), four sites must be updated in lockstep.

## Temporary host RAM pressure during per-parameter CPU move

The per-parameter `.data = .data.cpu()` loop creates a CPU copy of each tensor before the CUDA storage is freed. For a 7B-parameter model at fp16 (~14 GB), this temporarily requires 14 GB of host RAM in addition to whatever CPU memory is already used. On Colab free-tier T4 instances (typically 12–13 GB system RAM), this could trigger a host OOM or swap thrash that's harder to diagnose than a CUDA OOM. The risk is partially mitigated by the fact that T4 configs use the "3B" model (≈6 GB at fp16), but if a user overrides the LLM model to a larger variant, this becomes a real concern.
