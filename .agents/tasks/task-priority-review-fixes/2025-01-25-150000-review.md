# VRAM lifecycle, conditional audio, and prompt expansion for T4

This change addresses three priority tiers: preventing VRAM accumulation between extended-video segments (P0), reducing peak memory by sharing the video VAE and making audio generation optional (P1), and upgrading the inline LLM system prompt to the full production version (P2). The approach consolidates four separate VAE load calls into a single `_shared_vae` instance, wraps the entire audio pipeline in a `GENERATE_AUDIO` conditional, adds `keep_model_loaded` passthrough for batch LLM reuse, and replaces `imageio.mimsave(codec='libx264')` with a two-step ffmpeg encode for better h264 quality.

**Watch for:** The conditional-audio gate (`GENERATE_AUDIO=False`) creates a crash path â€” `LTXVSeparateAVLatent` and `LTXVConcatAVLatent` are still called unconditionally in Pass 2, but `catav` is only defined inside the `if _gen_audio:` branch (confirmed). The `save_frames_as_mp4` helper double-encodes every video through lossy compression (confirmed). The shared VAE stays resident across the entire generation pipeline including both sampling passes, which may not actually reduce peak VRAM compared to load-use-free (likely).

## High-level view

The audio-conditional gate wraps VAE loading, latent creation, AV concatenation, and audio decode inside `if _gen_audio:` blocks. But it leaves the Pass 2 pipeline (latent separation, re-concatenation, and the `catav` variable itself) unguarded, so setting `GENERATE_AUDIO=False` will crash with a `NameError` on `catav` or produce undefined behavior from splitting a video-only latent.

The shared VAE eliminates four redundant `VAELoader` calls (anchor, I2V, upscale, decode) and replaces them with a single load after UNet+LoRA. The tradeoff is that `_shared_vae` now lives across the entire pipeline â€” through both sampling passes â€” rather than being loaded and freed per-phase. On a 16 GB T4 where the UNet is also resident during sampling, holding the VAE simultaneously may offset the savings from avoiding reload overhead.

The `save_frames_as_mp4` helper first writes a temp file via `imageio.mimsave` (no codec specified, so imageio-ffmpeg picks a default â€” typically h264), then re-encodes it through ffmpeg with `-c:v libx264 -crf 19`. The intermediate file's codec becomes a quality floor for the final output â€” if imageio picks a lossy default, the pipeline applies lossy compression twice.

The LLM `keep_loaded` passthrough keeps the model on GPU between storyboard scenes and unloads after the last. If the loop raises an exception mid-way, the LLM remains loaded with no cleanup in the `except` block â€” on T4 with tight VRAM, this could prevent the video pipeline from starting.

## Audio-off crash path through Pass 2

When `_gen_audio=False`, the `else` branch at line ~5093 sets `combined_latent = _vid_lat_input` and `vae_audio = None`, but crucially `catav` (the `LTXVConcatAVLatent` node instance) is never instantiated â€” it's defined only inside `if _gen_audio:`. After Pass 1 sampling, the code unconditionally calls:

```python
# Line ~5210
ltxvsep    = NODE_CLASS_MAPPINGS["LTXVSeparateAVLatent"]()
s1         = ltxvsep.EXECUTE_NORMALIZED(av_latent=p1_av)
vid_lat_p1 = get_value_at_index(s1, 0)
aud_lat_p1 = get_value_at_index(s1, 1)
```

If Pass 1 received a video-only latent (no audio channels), `LTXVSeparateAVLatent` may either crash or return a zeroed/garbage audio tensor. Then:

```python
# Line ~5265
av_lat2 = catav.EXECUTE_NORMALIZED(
    video_latent=get_value_at_index(upsampled, 0),
    audio_latent=aud_lat_p1)
```

This raises `NameError: name 'catav' is not defined`. The entire Pass 2 pipeline â€” separation, upsampling concat, second sampling, final decode split â€” needs to be branched on `_gen_audio`. When audio is off, the video latent should flow directly without AV separation or concatenation.

## Double lossy encoding in save_frames_as_mp4

The function writes frames through two encoding stages:

```python
imageio.mimsave(_temp, [f for f in frames_np], fps=fps)  # Stage 1
subprocess.run(["ffmpeg", "-y", "-i", _temp, "-c:v", "libx264", ...])  # Stage 2
```

Stage 1 calls `imageio.mimsave` without specifying a codec. When writing to an `.mp4` extension, imageio-ffmpeg will typically default to h264 with its own quality settings. Stage 2 then decodes that compressed h264 and re-encodes with the desired CRF. This decode-then-encode cycle introduces generational loss â€” particularly visible as macro-blocking in areas with fine detail or motion.

The fix is to either pipe raw frames directly to ffmpeg (avoiding the intermediate file entirely) or explicitly tell imageio to write a lossless intermediate: `imageio.mimsave(_temp, frames, fps=fps, codec='rawvideo', pixelformat='rgb24')` and rename the temp to `.avi` or use a lossless codec like `ffv1`.

## Shared VAE residency during sampling

Prior to this change, the VAE was loaded fresh before each use (anchor encode, I2V conditioning, upsampling, decode) and freed immediately after. The new `_shared_vae` is loaded once at line ~4837 and freed at line ~5362 after audio decode. During both sampling passes the UNet (~5-8 GB for GGUF) and the VAE (~0.5-1 GB) are co-resident.

On T4 with 16 GB total, holding an extra GB through sampling could push the system into swap territory if the UNet, conditioning tensors, and intermediate latents already saturate VRAM. The removal of `del vae_for_anchor; aggressive_cleanup("VAE anchor done")` and similar per-phase cleanup calls means those VRAM dips no longer occur â€” the pipeline now has a higher sustained floor rather than peaks with valleys between phases.
