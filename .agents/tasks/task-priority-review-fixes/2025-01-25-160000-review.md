# VRAM lifecycle, conditional audio, and prompt expansion for T4 (v2)

This change consolidates VAE loading into a single shared instance, gates the entire audio pipeline behind `GENERATE_AUDIO`, replaces double-lossy `imageio.mimsave` → ffmpeg re-encode with a direct raw-frame pipe to ffmpeg, adds `keep_model_loaded` passthrough for batch LLM use in storyboards, upgrades the inline system prompt, and introduces `force_unload_all_models()` between extended-video segments.

**Watch for:** The ffmpeg pipe can deadlock if ffmpeg's stderr buffer fills before `proc.wait()` is called, since both stdout and stderr are `PIPE`'d but neither is drained during frame writes (likely). The `save_frames_as_mp4` fallback writes through imageio without an explicit codec — uncontrolled quality settings (possible). The expanded SYSTEM_PROMPT contains explicit adult content instructions sent to the LLM regardless of scene content (confirmed).

## High-level view

The audio-off crash path from v1 is fully addressed. Every Pass 2 AV operation — separation, concatenation, audio decode — is gated behind `if _gen_audio:`, with fallthrough paths that pass the video latent directly.

The `save_frames_as_mp4` function eliminates double lossy encoding by piping raw RGB24 frames to ffmpeg's stdin. On failure, it logs the exception before falling back to imageio — addressing both the double-encode and silent-failure concerns from v1. The subprocess management has a deadlock vector: `stdout=PIPE` and `stderr=PIPE` are set but neither is drained until after `proc.wait()`. For long videos, ffmpeg's stderr output could fill the OS pipe buffer (64 KB) and block both processes.

The shared VAE residency tradeoff is accepted by design. The inter-segment `force_unload_all_models()` call addresses VRAM accumulation between extended-video segments.

<summary>Issues (3)</summary>

1. **Potential pipe deadlock in save_frames_as_mp4** — Both `stdout` and `stderr` are set to `PIPE` but not drained during frame writes. If ffmpeg writes >64 KB to stderr (common for long encodes with progress output), ffmpeg blocks, Python blocks on `proc.stdin.write`, and the process deadlocks. Fix: use `proc.communicate()` after closing stdin, or set `stderr=subprocess.DEVNULL`, or drain stderr in a background thread.

2. **Fallback imageio encode has no quality control** — When the ffmpeg pipe fails, the fallback `imageio.mimsave(output_path, frames_list, fps=fps)` lets imageio pick its own codec and CRF (typically ~25, vs intended 19). On setups where ffmpeg invocation fails, this becomes the primary path with noticeably lower quality. Fix: pass `output_params=["-crf", "19"]`.

3. **Unconditional adult-content system prompt** — The expanded `SYSTEM_PROMPT` includes detailed undressing/erotic instructions sent to every prompt expansion call. For non-adult scenes, the LLM may inject inappropriate detail. Consider a `PROMPT_MODE` flag to select SFW vs NSFW prompt templates.

## Pipe deadlock in save_frames_as_mp4

The subprocess is created with `stdout=subprocess.PIPE, stderr=subprocess.PIPE`:

```python
proc = subprocess.Popen([
    "ffmpeg", "-y", ...
], stdin=subprocess.PIPE, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
for frame in frames_list:
    frame_bytes = np.ascontiguousarray(frame).astype(np.uint8).tobytes()
    proc.stdin.write(frame_bytes)
proc.stdin.close()
proc.wait()
```

During encoding, ffmpeg writes progress and diagnostic information to stderr. The Python process doesn't read from `proc.stderr` until after `proc.wait()` returns. If the stderr buffer fills (64 KB on Linux), ffmpeg blocks on its stderr write. Since Python is also blocked trying to write the next frame to stdin, both processes deadlock.

For typical short clips (a few seconds at 25 fps), the stderr output stays under the buffer limit. But for extended videos or when ffmpeg emits per-frame warnings, the buffer fills. The standard fix is `proc.communicate(input=all_frame_bytes)` (concatenate all frames first), a reader thread for stderr, or `stderr=subprocess.DEVNULL`.

The `proc.stderr.read()` after a non-zero returncode has a secondary issue: if the deadlock occurs, execution never reaches the returncode check. If ffmpeg exits early, `proc.stdin.write()` raises `BrokenPipeError`, which is caught by the broad `except Exception` and triggers the fallback with a logged message.

## Fallback quality gap

```python
except Exception as e:
    print(f"[save_frames_as_mp4] ffmpeg pipe failed: {e}")
    import imageio
    imageio.mimsave(output_path, frames_list, fps=fps)
```

When imageio writes to `.mp4`, it uses imageio-ffmpeg with default settings. The default CRF varies by version but is typically around 25 — noticeably more compressed than the intended CRF 19. Since this fallback activates when direct ffmpeg invocation fails (missing binary, permission issues), it may be the primary path on some setups. Adding `output_params=["-crf", "19"]` would maintain quality parity.

## System prompt content scope

The `SYSTEM_PROMPT` now contains 2,000+ characters of instructions specifically for adult/erotic content generation (undressing sequences, explicit body-part naming rules, erotic-cinema framing). These instructions are unconditional — they're sent to every LLM prompt expansion call regardless of whether the user's scene involves adult content. If the notebook is used in mixed-content workflows, a `PROMPT_MODE = "sfw" | "nsfw"` flag selecting the appropriate prompt template would prevent unintended content injection.
